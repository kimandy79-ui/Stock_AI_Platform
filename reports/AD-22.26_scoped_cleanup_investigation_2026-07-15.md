# AD-22.26 Closure — Setup-Type-Scoped Cleanup Investigation (NO-GO)

**Date:** 2026-07-15
**Scope:** Read-only investigation. No writes, no code changes, no rerun.
**Question:** can `consolidation_base`-only rows be cleaned and regenerated on
the 5 campaign dates without touching `breakout`/`pullback`/
`trend_continuation`'s already-matured `step4_analysis`/`step5_proposals`/
`outcome_tracking_queue`/`signal_outcomes` rows?

**Result: NO-GO.** Step 4 can genuinely be scoped to one setup type. Step 5
cannot — it is architecturally "once per `signal_date` across all setup
types" (per CLAUDE.md), reads *all* of a date's `step4_analysis` rows with no
`setup_type` filter, and writes with an INSERT-only path that has no
per-setup-type partial-regeneration mode. Any Step 5 rerun needed to get
`consolidation_base`'s real disposition/rank/RR necessarily mints fresh
`proposal_id`s for the other three setup types too, reproducing the exact
orphan+duplicate hazard on their `outcome_tracking_queue`/`signal_outcomes`
rows that this scoped approach was meant to avoid.

## 3. Why Step 4 can be scoped but Step 5 cannot

**Step 4 — scoping is real and already supported.**
`Step4SetupValidationEngine.run()` takes an optional `setup_configs: dict[str,
dict] | None` parameter (`step4_setup_validation_engine.py:413`); passing
`{"consolidation_base": cfg}` runs validation for only that one setup type.
Nothing else in the read/write path forces "all active configs" — that's
just the default when the caller passes `None`. A manual
`DELETE FROM step4_analysis WHERE signal_date = ? AND setup_type =
'consolidation_base'` followed by a scoped `run()` call would cleanly
regenerate only `consolidation_base`'s step4 rows, leaving the other three
untouched. This part of the ask is achievable.

**Step 5 — scoping is not real, despite an inviting-looking parameter.**
`ProposalEngine.propose()` also takes a `setup_configs` parameter
(`step5_proposal_engine.py:1076`), but it does **not** filter what gets read
or written:

- `_read_analyses()` runs `_SQL_READ_ANALYSES`, which is
  `... FROM step4_analysis a ... WHERE a.signal_date = ?`
  (`step5_proposal_engine.py:959-963`) — **no `setup_type` filter, no
  `run_id` filter.** It always reads every setup type's step4 rows for the
  date, regardless of what's in `setup_configs`.
- `_build_rows()` looks up each row's own config via
  `setup_configs.get(setup_type, {})` (`step5_proposal_engine.py:1276,1303`)
  — a **silent empty-dict fallback**, not an error. Pass a
  `consolidation_base`-only `setup_configs` dict and the engine will still
  process every `breakout`/`pullback`/`trend_continuation` row, just scored
  against an empty config instead of each type's real active thresholds/
  weights — a silent scoring-correctness bug, not a scoping mechanism. This
  route must not be attempted even as a shortcut.
- `propose()`'s write path (`self._write(db_role, rows)`) is **INSERT-only**
  — the orchestrator's own `_rescore_with_ai_review` docstring confirms this
  explicitly ("M15's `_write` is INSERT-only, so the first pass's rows are
  deleted before re-proposing rather than updated in place",
  `pipeline_orchestrator.py:1444-1446") and the code that relies on it
  (`_delete_step5_rows`) always clears the **whole date** first, never a
  setup-type slice.

Net effect: to get `consolidation_base`'s real post-rescale disposition/
rank/RR (the actual goal of §5.4/§5.5), Step 5 must be rerun — reading the
fresh `consolidation_base` step4 rows together with the untouched other
three types' step4 rows (this part is fine, since the read has no
`run_id`/setup_type filter, so it naturally picks up a mixed old+new step4
input correctly). But Step 5's own **output** cannot be partially replaced:
whatever step5_proposals rows already exist for the date must be deleted in
full before the fresh `propose()` call, or the old rows sit alongside a
brand-new full duplicate set for every setup type. That deletion mints new
`proposal_id`s for `breakout`/`pullback`/`trend_continuation` too, which
cascades into the same M16 orphan+duplicate mechanism identified in the
prior investigation (`tracking_id`/`outcome_id` are `uuid5` functions of
`proposal_id`, which is `uuid4()`-random per Step 5 write) — this time for
their 104+112+232=448 tracking rows and 33+22+58=113 outcome rows, exactly
the data this scoped approach set out to protect.

## 4. Go/No-Go

**NO-GO** on a setup-type-scoped cleanup that leaves `breakout`/`pullback`/
`trend_continuation`'s step5_proposals and M16 outcome rows untouched. The
blocker is architectural (Step 5's per-date, all-setup-types, INSERT-only
design — CLAUDE.md's own "Step 5 runs once per signal_date across all setup
types" rule is exactly the invariant that rules this out), not a matter of
missing SQL predicates. Achieving true setup-type isolation would require a
code change to `ProposalEngine.propose()` (a delete-and-reinsert or
merge/upsert mode scoped to one `setup_type`, preserving other types' rows
and `proposal_id`s) — out of scope for this investigation ("no code changes,
no writes, no rerun").

No delete statements are proposed here, since the scoped route does not
avoid the hazard it was meant to avoid.
