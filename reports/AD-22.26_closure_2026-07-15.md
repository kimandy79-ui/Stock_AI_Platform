# AD-22.26 Closure — Part A Idempotency Findings (Part B: NO-GO)

**Date:** 2026-07-15
**Scope:** Read-only investigation of whether re-running
`run_prod_pipeline.py --date <date> --resume-from market_regime_classification`
against the 5 campaign dates (2026-06-11, 06-18, 06-26, 07-02, 07-08) — which
already have `step3_candidates`/`step4_analysis`/`step5_proposals`/
`outcome_tracking_queue`/`signal_outcomes` rows from the original campaign —
is safe (overwrite) or unsafe (duplicate/orphan).

**Result: Part B did NOT proceed.** Two of the three sub-questions confirm
safety; the third does not, and it is load-bearing because `outcome_queue_creation`
and `outcome_processing` unconditionally execute on any
`resume_from=market_regime_classification` run (they sit at `STEP_NAMES`
indices 12–13, both `>= resume_index=8`) — there is no CLI flag or
orchestrator parameter to stop the run after `step5_proposals`.

## Q1 — step3/step4/step5: SAFE (delete-then-insert, confirmed)

`PipelineOrchestrator.run()` calls `cleanup_calculated_outputs_for_date()`
unconditionally, *before* any step executes
(`pipeline_orchestrator.py:538`, inside the same `try` block as step
execution). That method (`pipeline_orchestrator.py:808-859`) issues
per-signal-date `DELETE` statements inside one atomic transaction, scoped to
whichever steps `resume_from` will actually re-run:

```
_DELETE_STEP3: DELETE FROM step3_candidates WHERE signal_date = ?
_DELETE_STEP4: DELETE FROM step4_analysis   WHERE signal_date = ?
_DELETE_STEP5: DELETE FROM step5_proposals  WHERE signal_date = ?
```

`STEP_NAMES` order (`pipeline_orchestrator.py:74-90`) places
`market_regime_classification` at index 8, `step3_universal_eligibility` at 9,
`step4_setup_validation` at 10, `step5_proposals` at 11. With
`resume_from="market_regime_classification"` (`resume_idx=8`), the cleanup
predicate `resume_idx <= STEP_NAMES.index(step)` is `True` for step3/4/5 and
**`False` for `feature_calculation`** (index 7) — so `daily_features` is
correctly left untouched (features are unaffected by the rescale; only the
`consolidation_base` scorer changed). Re-running for an already-processed
date is a clean delete-then-regenerate for step3/4/5. No duplication risk
here.

## Q2 — outcome_tracking_queue / signal_outcomes: **NOT SAFE**

The task's premise ("keyed/idempotent per `(ticker, signal_date)` or
similar") does not match the actual key. `outcome_queue.py`'s own docstring
claims reruns are "silent no-ops" because inserts are idempotent on a
deterministic `tracking_id` — true in isolation, but the determinism chain
breaks under this specific rerun scenario:

- `tracking_id` = `uuid5(NAMESPACE_URL, f"outcome_tracking_queue:{proposal_id}:{horizon_bd}")`
  (`outcome_queue.py:243-249`)
- `outcome_id` = `uuid5(NAMESPACE_URL, f"signal_outcomes:{proposal_id}:{horizon_bd}")`
  (`outcome_queue.py:252-259`)
- Both are deterministic **functions of `proposal_id`** — but `proposal_id`
  itself is `str(uuid.uuid4())`, a fresh random UUID minted every time
  `step5_proposal_engine.py` writes a proposal row
  (`step5_proposal_engine.py:1552`), not a deterministic function of
  `(ticker, signal_date, setup_config_id)`.

Consequence for this rerun: Q1's cleanup deletes the original
`step5_proposals` rows (and their original `proposal_id`s) and
`step4/5_setup_step5` regenerates fresh proposals with **new** `proposal_id`s
for the same ticker/date. `outcome_queue_creation` then re-enqueues against
these new `proposal_id`s, producing `tracking_id`/`outcome_id` values that
are **different** from the originals — `ON CONFLICT DO NOTHING` /
`ON CONFLICT ... DO UPDATE` never fire because there's no key collision. Net
effect:

1. The **original** `outcome_tracking_queue`/`signal_outcomes` rows (keyed to
   the now-deleted `proposal_id`) are left in place as **orphans** — dangling
   references to a `step5_proposals` row that no longer exists.
2. A **new, parallel** set of `outcome_tracking_queue` rows (and eventually
   `signal_outcomes` rows as horizons mature) is inserted for the same
   ticker/signal_date/horizon combinations.

Confirmed no FK constraint would catch this: `outcome_tracking_queue.proposal_id`
and `signal_outcomes.proposal_id` are declared `VARCHAR NOT NULL` with no
`FOREIGN KEY` (`schema_manager.py:356-369`, `:373-398`) — DuckDB accepts the
orphaning silently.

This is genuine duplication + orphaning, not the identity no-op the module
docstring implies (that guarantee only holds for a retried run of the *same*
step5 output, e.g. a crash-and-retry within one invocation — not a
delete-and-regenerate rerun).

## Q3 — `_already_run` / `force_rerun` gate

`run()` queries `pipeline_runs` for a prior `success`/`success_with_warnings`
row for the date (`_SELECT_ALREADY_RUN`, `pipeline_orchestrator.py:166-170`)
before doing anything else. If found and `force_rerun` is not `True`, `run()`
returns `STATUS_FAILED` with `"run_date already succeeded"` and performs no
work at all (`pipeline_orchestrator.py:500-512`) — cleanup never runs, so this
gate is fail-safe, not a silent-overwrite risk. However, it means the
re-run command as literally specified in the task —

```
run_prod_pipeline.py --date <date> --resume-from market_regime_classification --skip-earnings-refresh
```

— **omits `--force-rerun`** and would exit 1 immediately for all 5 dates
(each already has a `success`/`success_with_warnings` row from the original
campaign). `--force-rerun` is a real, already-wired CLI flag
(`tools/run_prod_pipeline.py:107-111`) and must be added for Part B's command
to do anything.

## Go/No-Go: **NO-GO on Part B as specified**

Q1 and Q3 are clean. Q2 is not, and it's not avoidable by construction — the
orchestrator has no `stop_after`/`until` parameter and no CLI flag to end the
run at `step5_proposals`; `outcome_queue_creation`/`outcome_processing` are
unconditionally in the tail of any `resume_from<=8` run
(`pipeline_orchestrator.py:591-597`). Running Part B's command as written
would leave 5 dates' worth of orphaned `outcome_tracking_queue`/
`signal_outcomes` rows (referencing deleted `proposal_id`s) sitting alongside
a fresh duplicate set, corrupting the very outcome-tracking tables that
downstream diagnostics/`config_recommender` read.

## Proposed manual cleanup (for architect approval before any write)

Before re-running each of the 5 dates, explicitly delete the stale outcome
rows for that `signal_date`, mirroring the existing
`cleanup_calculated_outputs_for_date` pattern (same delete-then-regenerate
philosophy, just extended to two tables it doesn't currently cover):

```sql
DELETE FROM outcome_tracking_queue WHERE signal_date = ?;
DELETE FROM signal_outcomes        WHERE signal_date = ?;
```

Run once per campaign date, **before** invoking
`run_prod_pipeline.py --date <date> --resume-from market_regime_classification
--skip-earnings-refresh --force-rerun`. This makes the M16 re-enqueue collide
with nothing (clean slate) instead of orphaning+duplicating.

**Trade-off to flag explicitly:** this discards any `signal_outcomes` already
computed for these dates from real subsequent price action (the original
campaign ran 2026-06-11→07-08; some 5bd/10bd/20bd horizons for the later
dates may already be `outcome_status='complete'` with realized returns).
Nothing is unrecoverable — `outcome_processing` recomputes from
`daily_prices`, which is untouched — but note that `_step_process` is invoked
with `run_date=<the historical campaign date>` and its due-row filter is
`eval_date <= run_date` (`outcome_queue.py:135-141`), so horizons maturing
*after* the historical `run_date` won't resolve during Part B's re-run itself
— they'll sit `pending` until the next live daily prod run sweeps them
(the platform's normal daily job continues to run past 2026-07-08 per
[[project_state]]). Expect a temporary gap in `signal_outcomes` for these 5
dates between Part B's re-run and the next live daily pass, not permanent
data loss.

I have not made this deletion or run Part B — this needs an explicit
go-ahead per the coder note's own stop condition ("If overwrite is NOT
confirmed safe... stop and report... for architect review before any write
happens").
